BUILD ORDER
===========

1. Cycle list screen — `src/app/cycles/page.tsx` (replace placeholder)
   List all physical ponds (filter out `isVirtualPondType`) with their current cycle status:
   open (show `priorityCycleCode` + open date) or closed (show last close date / "אין מחזור
   פעיל"). Link each row to open/manage its cycle.
   Permission for *viewing this list* is NOT a simple `meetsRequirement()` call — the spec
   says "ניהול תחום, מנהל צופה, והנהלה" (DOMAIN_MANAGE, VIEW_ONLY, EXECUTIVE) but explicitly
   excludes OPERATIONS. Write a small inline check: `[AccessLevel.FULL_EDIT,
   AccessLevel.DOMAIN_MANAGE, AccessLevel.EXECUTIVE, AccessLevel.VIEW_ONLY].includes(level)`.

2. Cycle open / edit screen + API — e.g. `src/app/cycles/[pondId]/page.tsx` + form +
   `src/app/api/cycles/route.ts` (POST to open, PATCH to edit an open cycle)
   Fields: `priorityCycleCode` (optional; once set on a cycle it can be changed to another
   value but never cleared back to empty — enforce server-side), `openedAt` (date, cannot be
   in the future), `openNotes`.
   Validate server-side: pond must be physical; pond must not already have another open
   cycle (`hasOverlappingCycle`); if editing an existing open cycle's `openedAt` to a later
   date, block if `hasTransfersBefore(cycleId, newDate)` is true (would orphan existing
   transfer reports). `priorityCycleCode` is `@unique` in the schema — catch the Prisma
   P2002 error and return a friendly Hebrew message instead of a 500.
   Permission: `withModuleAccess("תפעול", AccessLevel.EXECUTIVE, ...)` (spec: "רק לבעלי
   הרשאות ניהול תחום והנהלה" → FULL_EDIT/DOMAIN_MANAGE/EXECUTIVE, i.e. ≤ EXECUTIVE).
   Audit: `writeAudit({ action: "open_cycle", entityType: "GrowthCycle", ... })`.

3. Cycle close screen + API — e.g. `src/app/cycles/[cycleId]/close/page.tsx` +
   `src/app/api/cycles/[id]/close/route.ts`
   Show a live fish-balance preview using `getCycleFishBalance(cycleId)` before the user
   confirms. If `priorityCycleCode` is still empty, require it now (same uniqueness/no-clear
   rule as above). Fields: `closedAt` (must be ≥ cycle.openedAt, ≤ now; block if
   `hasTransfersAfter(cycleId, newClosedAt)` is true), `closeNotes`.
   Per spec, the ±100 balance check is a **warning, not a hard block** — if
   `!withinTolerance`, show a clear red/amber message with the actual numbers but still let
   the user confirm and close. Store the balance snapshot in the audit `after` payload either
   way.
   Permission: same EXECUTIVE rule as open. Audit action `"close_cycle"`.

4. Transfers screen — `src/app/transfers/page.tsx` (replace placeholder) + API routes under
   `src/app/api/transfers/`
   3-part flow per spec: (a) pick source pond + date → resolve the pond's active
   `GrowthCycle` automatically (error clearly if none open); (b) pick transfer type — קניה /
   דילול / פירוק / שיווק are one family on `FishTransferHeader.transferType`, תמותה
   (mortality) is reported separately even though it's stored the same way; (c) manage
   `FishTransferDetail` rows under the header (means, strain, count, avg weight, dest pond,
   time, population code). Header `status` defaults `"טיוטא"` → `"הסתיימה"`; detail `status`
   defaults `"טיוטא"` → `"סגירה"`. Support saving a draft and resuming it later (don't force
   a single-shot submit).
   RBAC ASSUMPTION (flag in a code comment + final report — spec doesn't state this
   section's level explicitly): default to `AccessLevel.OPERATIONS` (day-to-day data entry
   fits "תפעול שוטף"). Easy to tighten later once confirmed.
   Audit: `"create"`/`"update"` on entityType `"FishTransferHeader"`/`"FishTransferDetail"`.

5. Basket-weighing modal (not a standalone page) — triggered only from a דילול/פירוק/שיווק
   transfer detail row, writes `FishWeighingHeader` (with `transferDetailId` set) +
   `FishWeighingBasketDetail` rows (`emptyWetWeight`, `weightWithFish`, `fishCount` per
   basket, unique on `[headerId, basketSeq]`). Compute and display: avg fish weight per
   basket (`(weightWithFish - emptyWetWeight) / fishCount`) and the weighted total across all
   baskets in the session.
   Permission: same OPERATIONS assumption as #4.

6. Field / net weighing screens — `src/app/weighings/page.tsx` (replace placeholder)
   Two flows under one page (tabs or a type switch is fine): שקילות שטח (routine field
   sampling, no fixed schedule enforcement needed yet — just record `date`, `pondId`/
   `cycleId`, `tankId` optional, basket rows same shape as above) and שקילות רשת
   (net-sampling tied to a transfer, similar shape but `transferDetailId` set, no
   `tankId`/basket modal — check the schema fields actually present on
   `FishWeighingHeader`/`FishWeighingBasketDetail` before assuming field names).
   Permission: OPERATIONS (same assumption, same flag).
