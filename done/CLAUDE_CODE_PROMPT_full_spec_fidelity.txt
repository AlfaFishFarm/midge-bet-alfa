SECTION A — Transfers: weight-entry model inversion for דילול/פירוק/שיווק (HIGHEST PRIORITY —
data integrity, feeds מאזן מסה/inventory)

Spec page 18 (exact, paraphrased from mockup + text): for דילול/פירוק/שיווק, the field flow is the
OPPOSITE of קניה:
  - משקל ממוצע (גרם) → READ-ONLY, must equal the average computed from that row's basket-weighing
    session (BasketWeighingModal). Page 21 confirms: "המשקל הממוצע יישמר ברשומת שקילת הטנק (ראה
    מסכי שקילות)".
  - משקל כולל (ק"ג) → MANUAL entry (from the tank scale) — this is the field the user actually
    types into for these 3 types.
  - מספר דגים → COMPUTED = round(משקל כולל * 1000 / משקל ממוצע). NOT manually entered.
  - קניה stays as currently implemented: fishCount + avgWeightGrams are the manual inputs (from
    the supplier's report), total weight is the only computed field.

Current state (confirmed by reading the code 2026-06-21):
  - `src/app/api/transfers/[id]/details/[detailId]/route.ts` `patchDetailSchema` allows free-form
    editing of `avgWeightGrams`/`fishCount` for every transfer type uniformly — no per-type branch.
  - `prisma/schema.prisma` `FishTransferDetail` has no `totalWeightKg` field at all.
  - `src/app/transfers/[id]/BasketWeighingModal.tsx` computes `overallAvg`/`totalNet`/`totalFish`
    per weighing session but never PATCHes them back to the parent `FishTransferDetail` row — so
    even if avg weight were made read-only, nothing currently populates it from basket weighing.

Required changes:
  1. Add `totalWeightKg Float?` to `FishTransferDetail` (migration).
  2. In `src/app/transfers/[id]/BasketWeighingModal.tsx`, after a weighing session is saved, PATCH
     the parent detail row's `avgWeightGrams` with the session's computed average (and consider
     whether fishCount/totalWeightKg should also sync at that point, vs. staying purely
     user-entered on the detail row — match the spec's "see weighing screens" wording: avg weight
     is the ONLY field driven by basket weighing; total weight is a separate manual tank-scale
     entry).
  3. In the transfer detail row UI (`TransferDetailManager.tsx`) and its PATCH validation
     (`route.ts` above), branch on `header.transferType`:
     - קניה: fishCount + avgWeightGrams editable, totalWeightKg computed+displayed, read-only.
     - דילול/פירוק/שיווק: totalWeightKg editable, avgWeightGrams read-only (sourced from basket
       weighing; show "—" or a "טרם נשקל" placeholder until a weighing exists for that row),
       fishCount computed+displayed, read-only.
     - תמותה: see Section E — gets its own screen, same weight-model as דילול per spec page 19's
       mockup (avg weight + fish count both shown per row, total weight computed — re-check the
       page 19 mockup columns when implementing: סוג דג / משקל ממוצע (גרם) / מספר דגים / משקל כולל
       (ק"ג) / שעת ההרשמה / סיבת תמותה — closer to קניה's input pattern than דילול's, confirm
       against the mockup, don't assume).
  4. Server-side: reject PATCHes that try to set a read-only field for that transfer type (don't
     rely on UI disabling alone — same principle as the existing RBAC pattern in this codebase).
