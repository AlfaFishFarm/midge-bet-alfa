"בלחיצה על כפתור ביצוע פעולה, ולאחר בדיקת התנאים - להוציא הודעת אישור שמסכמת את הפעולה. אם

(= on clicking the action button, after validating: show a confirmation message summarizing the

confirm, the dialog closes and the screen returns to the operations screen.)

though it looks similar to what exists today:
  - The five entry screens currently show plain inline red text
    (`{error && <p className="text-red-600...">}`) with NO close button — this does NOT satisfy

  - This banner must appear both for client-side validation failures (before the confirm dialog

WHAT TO BUILD
================
For each of the entry screens listed above:
1. Split the current single-step "validate + immediately POST/PATCH" handler into two steps:
   a. On submit: run client-side validation as today (required fields, date ranges, etc.). If
      invalid, show the existing red inline error — no confirm dialog needed for plain field
      validation, only once the form is otherwise ready to save.
   b. If valid, open a confirmation modal/dialog (reuse the project's existing modal pattern from
      BasketWeighingModal.tsx if one exists, or build a small shared `ConfirmDialog` component in
      src/components/ if no shared modal exists yet) summarizing exactly what will be written:
      - Cycle open/update: pond name, the computed קוד מחזור (date-only, per
        CLAUDE_CODE_PROMPT_cycle_app_code.txt), מחזור פריוריטי (or "לא הוזן" if empty), תאריך
        פתיחה, הערות. Title: "אישור פתיחת מחזור גידול" or "אישור עדכון פרטי מחזור" depending on
        isEdit.
      - Cycle close: same fields plus the fish-balance summary that's already computed and shown
        inline today in CloseForm.tsx (balance.incoming/outgoing/mortality/difference) — carry
        that same block into the confirm dialog rather than duplicating logic. Title: "אישור
        סגירת מחזור גידול".
      - Transfer: transfer type, pond, date, קוד מחזור, and a compact list of the tank rows about
        to be saved (fish type, weight, count per row) with totals. Title depends on transfer
        type, e.g. "אישור העברת דילול".
      - Weighing: date/time, weighing type, pond, tank, קוד מחזור, basket rows with totals. Title
        "סיכום שקילה".
   c. Only on explicit confirm inside that dialog does the existing fetch(POST/PATCH) call fire.
      Cancel/close just dismisses the dialog and returns to the editable form, no request sent.
   d. If the save request itself fails (server-side validation, network error), close the
      confirm dialog and show the existing red inline error on the form — don't silently fail
      inside the dialog.
2. Keep all existing validation logic, RBAC, and audit-log calls exactly as they are — this is
   purely an added confirmation step in the UI flow, not a change to what gets validated or
   saved.
3. Style: green/teal for the confirm button (matches BTN_PRIMARY conventions already in these
   files), keep Hebrew RTL layout, reuse INPUT/BTN_PRIMARY-style constants already defined per
   file rather than inventing new ones.

GUARDRAILS
============
- Don't change any validation rule, RBAC check, or audit-log payload — only the save/UX flow.
- Don't add this pattern to any list/display screen.
- Don't build a generic "are you sure" browser confirm() — the spec wants a styled in-app dialog
  that shows the actual values about to be saved, not a generic yes/no prompt.
- If a shared ConfirmDialog component doesn't exist yet, create one reusable component rather
  than copy-pasting modal markup into all five screens.
